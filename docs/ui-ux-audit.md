# UI/UX Audit

Audit date: 2026-06-04  
Audited commit: `e5d2ed3`

## Executive summary

This repository ships a static multi-page web application for Tibia Warzone schedules, rankings, open-house listings, and maintainer operations. The public experience is built with hand-authored HTML, one global stylesheet, and page-specific vanilla JavaScript controllers. Data is generated by Python scripts and deployed directly to GitHub Pages.

The strongest product risks are not visual polish issues. They are interaction and accessibility defects that reduce user control and clarity:

- Primary navigation compresses below Apple HIG touch-target and legibility expectations on small screens.
- Localization is only partially implemented. Several pages expose language controls that do not fully work, translations are incomplete, and the document language metadata does not track the displayed language.
- The two modal experiences inspected during runtime testing do not meet basic keyboard and focus-management expectations.
- The deployment artifact includes the full raw `data/` tree and large market datasets without a build step, minification pass, or payload boundary.
- Current repository documentation still describes admin open-house editing behavior that the code no longer exposes.

## Project overview

| Area | Verified implementation |
| --- | --- |
| Project type | Static multi-page website with generated JSON data |
| Public platforms | GitHub Pages static web delivery |
| Frontend stack | Plain HTML, global CSS, vanilla JavaScript modules loaded with `<script defer>` |
| Styling approach | Single global stylesheet at `assets/styles.css` |
| UI component system | No framework component layer; DOM strings and direct element mutation in page controllers |
| State management | Page-local mutable state plus `localStorage` (`assets/app.js`, `assets/ranking.js`, `assets/open-houses.js`, `assets/admin.js`, `assets/shared.js`) |
| Routing architecture | File-based static routes plus query-string state (`index.html`, `ranking.html`, `world.html`, `open-houses.html`, `bigfoot.html`, `admin.html`) |
| Localization approach | Per-page JavaScript dictionaries with a shared saved language key in `localStorage`; no central i18n build step |
| Python/runtime dependencies | `requirements.txt` contains `numpy`; scripts are executed directly with `python3` |
| Test framework | Python `unittest` |
| Build system | No bundler, no transpiler, no package-based frontend build step |
| Deployment | GitHub Pages workflow copies source files and `data/` directly into `_site` |
| Documentation structure | `README.md`, `Expected_Return_Explanation.md`, and `docs/*.md` |

### Verified user-facing surfaces

- `index.html`: world overview, filters, timezone selector, schedule planner, print-list modal, summary, world cards.
- `ranking.html`: ranking filters, summary, ranking table.
- `world.html`: world summary, manual schedules, bazaar card, market prices, history card, market-item modal.
- `open-houses.html`: world overview mode and world-detail mode selected by `?world=...`.
- `bigfoot.html`: static reference page with outbound quest links.
- `admin.html`: maintainer editor for manual schedules and tracked market items, plus PR workflow UI.

## Audit methodology

1. Repository discovery from source, tests, workflows, manifests, and runtime behavior.
2. Static inspection of all public HTML entry points and their corresponding JavaScript controllers.
3. Verification of declared Python dependencies and repository validation scripts.
4. Browser inspection of rendered UI using a local `python3 -m http.server 4174` instance and runtime interaction checks on desktop and mobile-sized viewports.
5. Cross-check of current documentation claims against current code behavior.

### Tool limitations

- No frontend package manifest, Storybook config, visual-regression suite, lint config, or automated accessibility test harness was present.
- Live TibiaMarket sampling was limited by a name-resolution failure during the bounded fetch command.
- `scripts/update_open_houses.py` could not be fully executed without `GITHUB_TOKEN` and `GITHUB_REPOSITORY`.
- `scripts/update_data.py` execution was partially limited by repeated upstream `503` responses from `api.tibiadata.com`.

## Visual design findings

### UX-01
Title: The visual system is visually distinctive but not platform-adaptive  
Severity: Medium  
Category: Visual design  
Evidence:
- File: `assets/styles.css`
- Function/component: root theme tokens and global theme rules
- Lines or code references: `assets/styles.css:12-30`, `assets/styles.css:52-79`, `assets/styles.css:105-112`
- Command output, test result, or runtime evidence: All inspected public pages rendered the same dark glass treatment; no light-theme branch or theme switcher was found in the code.
Impact:
The interface favors brand expression over deference and system familiarity. The strong Instagram-style gradient and dark-only palette make the product feel custom-branded, but they reduce platform fit and limit adaptability for users who expect lighter or more neutral presentation.
Apple HIG principle involved:
Clarity, deference, and platform consistency.
Recommendation:
Separate brand accents from foundational surface and text tokens, and add a lighter neutral presentation path or system-driven color adaptation.
Acceptance criteria:
- Base surfaces and typography remain legible without the gradient overlay carrying the hierarchy.
- A light or system-adaptive palette exists in code and can be verified from source.
- Accent color is used intentionally for emphasis rather than as the dominant surface treatment.
Suggested implementation approach:
Introduce a neutral surface token set, move the saturated gradient treatment to selective accents or headers, and add a second token block driven by `prefers-color-scheme` or a product theme setting.

### UX-02
Title: Web fonts are loaded with render-blocking CSS `@import`
Severity: Medium  
Category: Visual design / performance  
Resolution: Completed on 2026-07-16. All shipped HTML entry points now establish
early Google Fonts connections and load the font stylesheet directly before the
local stylesheet. Repository validation rejects CSS `@import` regressions and
missing entry-point font markup.
Evidence:
- File: `assets/styles.css`
- Function/component: global font import
- Lines or code references: `assets/styles.css:12`
- Command output, test result, or runtime evidence: No frontend build system or asset pipeline exists in the repository; the font request is therefore shipped as authored.
Impact:
`@import` delays stylesheet completion and adds an extra network dependency to first paint. This is especially visible on a static site that otherwise has no build-time optimization pass.
Apple HIG principle involved:
Performance and clarity.
Recommendation:
Move font loading to `<link rel="preconnect">` and `<link rel="stylesheet">` in the document head, or self-host the font assets.
Acceptance criteria:
- No CSS `@import` is used for webfont delivery.
- Initial render remains legible with system fonts while custom fonts load.
Suggested implementation approach:
Replace the import with explicit head links on each public page, or package the font files locally and declare them with `@font-face`.

## Layout and spacing findings

### UX-03
Title: Core interactive controls fall below Apple HIG touch-target guidance
Severity: High  
Category: Layout and spacing / accessibility  
Progress: Filter pills, the language trigger and options, and the GitHub icon
now maintain a 44px minimum target at every viewport as of 2026-07-16. The
remaining topbar route links, search, and timezone controls still require
verification and remediation before this finding is resolved.
Evidence:
- File: `assets/styles.css`
- Function/component: topbar links, language button, search input, timezone select, filter pills
- Lines or code references: `assets/styles.css:196-232`, `assets/styles.css:268-296`, `assets/styles.css:317-330`, `assets/styles.css:539-590`, `assets/styles.css:630-695`
- Command output, test result, or runtime evidence: Browser inspection on `index.html` recorded topbar link height `24px`, language button `34x34`, search input `40px`, timezone select `40px`, and filter pills `40px`; mobile inspection at `390x844` recorded topbar links at `21px` height and filter pills at `36px`.
Impact:
These dimensions reduce touch accuracy, increase accidental taps, and make the primary navigation harder to use on phones. The issue is systematic, not isolated.
Apple HIG principle involved:
Touch targets, accessibility, and clarity.
Recommendation:
Raise all primary tap targets to at least `44x44` CSS pixels and avoid reducing target size at narrow breakpoints.
Acceptance criteria:
- Topbar links, language button, print action, search, timezone selector, and filter pills all meet a `44x44` minimum interactive area.
- Mobile view no longer relies on reduced font size and padding to fit the nav.
Suggested implementation approach:
Increase min-heights and horizontal padding, convert the topbar to a more resilient small-screen pattern, and separate visual density from hit-area size by using transparent padding.

### UX-04
Title: Small-screen navigation is solved by compression rather than adaptation
Severity: Medium  
Category: Layout and spacing / navigation  
Evidence:
- File: `assets/styles.css`
- Function/component: responsive topbar rules
- Lines or code references: `assets/styles.css:205-214`, `assets/styles.css:268-309`
- Command output, test result, or runtime evidence: Mobile inspection at `390x844` showed the Bigfoot route collapsed to `Quest`, nav links at `21px` height, and very narrow link widths such as `37px` for `Home`.
Impact:
The navigation remains technically visible but loses scannability and tap comfort. Abbreviating one destination while shrinking all others weakens information scent instead of adapting the pattern.
Apple HIG principle involved:
Adaptivity, clarity, and navigation.
Recommendation:
Replace small-screen compression with a navigation pattern designed for constrained width, such as a disclosure menu or a stacked secondary route list.
Acceptance criteria:
- Route labels remain understandable on narrow screens.
- No top-level route is reduced to a cryptic one-word abbreviation solely to fit the bar.
- Navigation remains comfortably tappable without overflow.
Suggested implementation approach:
Promote the most-used routes into the top level, move secondary destinations into a menu, and stop using `white-space: nowrap` plus padding reduction as the main responsive strategy.

Severity: Medium  
Category: Layout and spacing / accessibility

responsive CSS Grid with stable row-major DOM order. Runtime column wrappers,
card measurement, resize-triggered full renders, and duplicated masonry code

Evidence:
- File: `assets/app.js`
- Function/component: `applyMasonry`
- Lines or code references: `assets/app.js:1980-2008`
- File: `assets/open-houses.js`
- Function/component: `applyMasonry`
- Lines or code references: `assets/open-houses.js:241-255`

Acceptance criteria:
- Reading order matches visual order in keyboard traversal and screen-reader output.
- Card layout does not require DOM redistribution after render.
Suggested implementation approach:
Use CSS grid or multicolumn layout for presentation while preserving a stable source order, or accept a simpler single-column list when accessibility is prioritized.
